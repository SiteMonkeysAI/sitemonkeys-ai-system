<h2>Context</h2>
<p>PR #806 successfully fixed 2 of 3 targeted failures:</p>
<ul>
<li>✅ Failure 2 (News price queries): External lookup fires, directs to Google Finance/Yahoo Finance</li>
<li>✅ Failure 3 (Greenland freshness): Freshness marker override triggers SEMI_STABLE correctly</li>
<li>❓ Failure 1 (PDF): Untested in this session (no PDF uploaded)</li>

<p>However, comprehensive production testing with real queries and Railway log analysis revealed that the remaining issues are NOT isolated bugs — they are <strong>two systemic engineering problems</strong> that create the illusion of regression every time a point fix is applied.</p>
<h2>The Two Root Problems</h2>
<h3>Problem 1: Non-Deterministic Output Paths</h3>
<p>Business boilerplate, price response formatting, and other guardrails are implemented as <strong>prompt-level guidance</strong> — instructions injected into the system prompt asking GPT-4 to comply. This is unreliable because:</p>
<ul>
<li>Different code paths (Eli vs Roxy, standard vs external-data) may bypass the injection</li>
<li>GPT-4 may ignore prompt instructions on any given call</li>
<li>Post-generation appenders (survival analysis, risk assessment) run unconditionally</li>
</ul>
<p><strong>Evidence from production:</strong></p>
<ul>
<li>Walmart stock query → Roxy says "the specific current stock price of Walmart is not provided" (violates "never say price not given" clamp)</li>
<li>Walmart stock query → "⚠️ Critical Risks I See" + "Business Survival Analysis: Runway Impact: undefined" appended</li>
<li>Colombia news query → "💰 Business Survival Analysis: Critical Dependencies: 1 identified"</li>
<li>Hardscaping document summary → business boilerplate + risk analysis appended to keyword list analysis</li>
</ul>
<p><strong>The principle:</strong> When the system requires deterministic behavior (no business boilerplate on non-business queries, no "price not given" on RSS-only responses), it must be enforced <strong>in code after generation</strong>, not by asking the model to comply.</p>
<h3>Problem 2: Non-Persistent Document State</h3>
<p>The <code>extractedDocuments</code> Map is stored in process memory. Railway deployments can involve cold starts, multiple instances, or process restarts. An in-memory Map is not reliable shared state.</p>
<p><strong>Evidence from production:</strong></p>
<ul>
<li>First 4 document upload attempts: <code>extractedDocuments Map size: 0</code> → "I don't have any document loaded"</li>
<li>5th attempt suddenly works: <code>✅ Document loaded: pasted_document.txt (731 tokens)</code></li>
<li>The DOCX uploads eventually succeed, but the pattern is non-deterministic</li>
</ul>
<p><strong>The principle:</strong> Document state must be stored in the same persistence layer as everything else (PostgreSQL/KV). A process-local Map that vanishes on restart is architecturally wrong for a deployed service.</p>

<h3>FAILURE C: Document Upload State Not Persistent [P0]</h3>
<p><strong>Symptom:</strong> First 3-4 document uploads fail with <code>extractedDocuments Map size: 0</code>. Documents eventually appear after repeated uploads.</p>
<p><strong>Log evidence:</strong></p>
<pre><code>22:44:43 [HANDOFF:DOCUMENT-LOAD→CONTEXT] ❌ No document found
22:44:43 extractedDocuments Map size: 0
...
22:46:50 [HANDOFF:DOCUMENT-LOAD→CONTEXT] ✅ Document loaded: pasted_document.txt
</code></pre>
<p><strong>Root cause:</strong> <code>extractedDocuments</code> is an in-memory JavaScript Map. On Railway:</p>
<ul>
<li>Process restarts clear the Map</li>
<li>Multiple instances don't share memory</li>
<li>Cold starts mean the Map is empty until explicitly populated</li>
<li>Even within a single instance, there may be timing issues between the upload endpoint writing to the Map and the orchestrator reading from it</li>
</ul>
<p><strong>Expected behavior:</strong> Every successful upload should be immediately retrievable on the next query, regardless of process state. Documents should persist for the session duration.</p>
<p><strong>Investigation approach (Phase 1):</strong></p>
<ol>
<li>Find where <code>extractedDocuments</code> is declared (likely <code>server.js</code> or an upload handler)</li>
<li>Trace the write path: upload endpoint → processFile() → where does extracted text go?</li>
<li>Trace the read path: orchestrator.js → where does it look for documents?</li>
<li>Determine if these are the same Map instance or different references</li>
<li>Check if Railway is running multiple instances (check <code>RAILWAY_REPLICA_ID</code> or similar)</li>
<li>Post findings as comment</li>
</ol>
<p><strong>Candidate fix:</strong></p>
<ul>
<li>Store extracted document content in PostgreSQL (or Upstash KV if available) keyed by <code>user_id + session_id + doc_id</code></li>
<li>Read from DB/KV in the orchestrator instead of in-memory Map</li>
<li>Set TTL (e.g., 24 hours) so documents auto-expire</li>
<li>This also fixes the PDF issue: even if pdf-parse works perfectly, extracted text stored in a process-local Map will vanish</li>
</ul>

<p><strong>Symptom:</strong> Query classifier crashes when processing large document context.</p>
<p><strong>Log evidence:</strong></p>
<pre><code>[QUERY_CLASSIFIER] Error getting embedding: BadRequestError: 400 This model's maximum context length is 8192 tokens, however you requested 8913 tokens
</code></pre>
<p><strong>Root cause:</strong> The hardscaping document (5,594 words) plus the query exceeds the 8192 token embedding model limit. The query classifier is passing the full query + document context to the embedding API without truncation.</p>
<p><strong>Expected behavior:</strong> Query classifier should truncate or exclude document content before generating embeddings. The classification should be based on the user's query, not the full document.</p>
<p><strong>Candidate fix:</strong> Before calling the embedding API in the query classifier, truncate the input to max 7500 tokens (leaving headroom). Strip document content first — classify based on the user message only.</p>
