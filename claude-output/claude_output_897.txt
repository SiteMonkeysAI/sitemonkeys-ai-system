## Changes Made

**File modified:** `api/core/intelligence/externalLookupEngine.js`

### What was done

**Problem:** When a user sends a conversational query like "Seems like the fed has something going on lately", `cleanNewsQuery()` strips stop words but leaves conversational noise words ("Seems", "like", "something"). The cleaned result "Seems like fed something lately" has 4 "meaningful" words by the existing `#885` viability check (which only filters time fillers like "lately"), so the institution-name fallback doesn't trigger. This bad query is sent to news APIs, which return 0 results or noise. When 0 results are returned, the orchestrator already injects a disclosure (from the #885 fix), so the silent-failure disclosure path is handled.

**Root gap:** The `#885` viability check inside `cleanNewsQuery()` only filters `TIME_FILLER_WORDS` (`lately`, `recently`, etc.), leaving conversational noise words (`seems`, `like`, `something`) to inflate the meaningful word count past the 2-word threshold. This prevented the `Federal Reserve latest news` fallback from firing.

### Fix implemented

**Added `ensureQueryViability(cleanedQuery, originalQuery)`** — a new function placed **after** `cleanNewsQuery()`, following the constraint that the fix should be a post-cleaning viability check that runs after the existing function completes. It uses an **expanded `VIABILITY_NOISE` filter** that includes conversational filler words (`seems`, `like`, `something`, `anything`, etc.) in addition to time fillers.

**Logic:**
1. Counts words in the cleaned query that are NOT in the expanded noise list
2. If fewer than 2 truly meaningful words remain AND the **original** query contains a known institution name (via `INSTITUTION_PATTERN`), returns `"{ExpandedName} latest news"` (e.g. `"Federal Reserve latest news"`)
3. Otherwise returns the cleaned query unchanged

**Wired into all 5 news sources** by replacing `cleanNewsQuery(query)` with `ensureQueryViability(cleanNewsQuery(query), query)` at each call site:
- NewsAPI (`buildUrl`, line 515)
- Serper (`fetchData`, line 553)
- TheNewsAPI (`buildUrl`, line 599)  
- Google News RSS (`buildUrl`, line 627)
- GDELT (`buildUrl`, line 651)

### What was NOT changed

- `cleanNewsQuery()` core logic (steps 1–7) — unchanged per constraint
- The existing `#885` viability check inside `cleanNewsQuery()` — left in place (handles the simple "fed" alone case, no interference)
- The orchestrator's silent-failure disclosure injection (`#885` fix at line 4053-4058) — already present, no changes needed
- No test files modified

### Regression safety

Known working queries verified not to regress:
- `"Did Saudi Arabia make a big commitment to the US?"` → cleaned to `"Saudi Arabia commitment"` (3 meaningful words) → `ensureQueryViability` returns unchanged ✓
- `"What is the FDA doing about the Ozempic situation?"` → cleaned to `"FDA Ozempic situation"` (3 meaningful words) → unchanged ✓  
- `"What is NATO doing right now?"` → `cleanNewsQuery` already returns `"NATO latest news"` (via #885) → `ensureQueryViability` sees 3 non-noise words → unchanged ✓
- `"What are the names of my children?"` → no lookup triggered ✓
- Photosynthesis / general knowledge → no lookup triggered ✓
