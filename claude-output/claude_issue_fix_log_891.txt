### Fix 1: MEM-007 — Importance-Based Priority Ranking

**File modified:** `api/services/semantic-retrieval.js`

**Location:** Inside `calculateHybridScore()`, after the non-boosted `score = Math.min(score, 1.0)` cap.

**Root cause:** `calculateHybridScore()` never used `memory.relevance_score` even though:
- `analyzeContentImportance()` already scored allergy at 0.95, ice cream at ~0.50
- `storeCompressedMemory()` stored those as `relevance_score` in the DB
- The DB SELECT already returned `relevance_score` in results

**What was added:**
```javascript
if (memory.relevance_score && memory.relevance_score >= 0.85) {
  const importanceBoost = (memory.relevance_score - 0.85) * 3.0;
  score += importanceBoost;
  console.log(`[IMPORTANCE-BOOST] Memory ${memory.id}: ...`);
}
```

**Tier separation:**
- Non-boosted cap: ≤1.0
- Importance-boosted (allergy 0.95): reaches ~1.30 (+0.30)
- Safety-boosted (existing tier): ≥2.0

### Fix 2: CMP2 — Multi-Word Names Without Special Characters

**File modified:** `api/memory/intelligent-storage.js`

**Location:** Compression prompt Rule 3 and contacts example.

**Root cause:** Rule 3 was titled "PRESERVE INTERNATIONAL NAMES WITH ALL SPECIAL CHARACTERS" — GPT-4o-mini could interpret this as only requiring preservation when special chars are present. "Zhang Wei" has NO diacritics or hyphens, so GPT could drop "Wei" without technically violating the old rule.

**What was changed:**
- Rule 3 header: now "PRESERVE ALL PROPER NAMES EXACTLY AS WRITTEN"
- Added explicit sub-bullets: "East Asian names: 'Zhang Wei', 'Li Ming' → keep BOTH words"
- Added new contacts example with "Zhang Wei, Björn Lindqvist, José García" and explicit note that "Zhang Wei" requires BOTH words

**E-004 safety:** The "diacritics" keyword still appears in Rule 3 sub-bullets — satisfies the existing OR check in code-guards.
