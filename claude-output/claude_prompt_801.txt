## Bug 1: `memoryText.match is not a function` — CRASH (P0)

**Error:**
```
TypeError: memoryText.match is not a function
    at #extractNumericalData (orchestrator.js:4387:40)
    at #buildContextString (orchestrator.js:4603:93)
    at #routeToAI (orchestrator.js:3833:53)
```

**Root cause:** The `#extractNumericalData` method calls `memoryText.match()` expecting a string, but `context.memory` is an **array** of memory objects, not a string. Before the context.sources fix, this code path was never reached because `context.sources?.hasMemory` was always undefined and the memory injection block was skipped. Now that `context.memory` correctly evaluates as truthy, the code enters the memory block and crashes.

**Fix:** At the top of `#extractNumericalData(memoryText)`, add type handling:
```javascript
#extractNumericalData(memoryText) {
    // Handle array input (memory context is an array of objects)
    if (Array.isArray(memoryText)) {
        memoryText = memoryText.map(m => typeof m === 'string' ? m : JSON.stringify(m)).join('\n');
    }
    if (typeof memoryText !== 'string') {
        memoryText = String(memoryText || '');
    }
    // ... rest of existing method
```

Also check `#buildContextString` around line 4603 — wherever it calls `#extractNumericalData`, ensure the argument being passed is converted to a string first. There may also be other places in `#buildContextString` that do `.match()`, `.includes()`, or `.length` on `context.memory` assuming it's a string when it's actually an array. Search for ALL uses of `context.memory` inside `#buildContextString` and ensure they handle arrays.

## Bug 2: Documents never reach orchestrator — `extractedDocuments Map size: 0`

**Evidence (every single request):**
```
[HANDOFF:DOCUMENT-LOAD→CONTEXT] ❌ No document found
[HANDOFF:DOCUMENT-LOAD→CONTEXT] extractedDocuments Map size: 0
[ORCHESTRATOR] [DOCUMENTS] No document found in storage
[ORCHESTRATOR] [DOCUMENTS] No document available
```

**Root cause:** The frontend shows "Uploading 1 file(s) for analysis..." but the `extractedDocuments` Map in the orchestrator is always empty. The upload endpoint (`api/upload-file.js` or `api/upload-for-analysis.js`) either:
1. Fails to extract text from the uploaded file, OR
2. Extracts text but stores it somewhere the orchestrator doesn't look, OR
3. The Map key doesn't match what the orchestrator queries

**Fix approach:** 
1. Search for where `extractedDocuments` is populated (likely in the upload route or a middleware)
2. Search for where `extractedDocuments` is read (in `#loadDocumentContext` in orchestrator.js)
3. Verify the Map key used for storage matches the key used for retrieval (usually `chatId` or `userId`)
4. Add logging at the upload endpoint: `console.log('[UPLOAD] Stored document for key:', key, 'content length:', content.length)`
5. Add logging at retrieval: `console.log('[DOC-LOAD] Looking up key:', key, 'Map size:', extractedDocuments.size, 'Map keys:', [...extractedDocuments.keys()])`

The files to check are:
- `api/upload-file.js`
- `api/upload-for-analysis.js`  
- `server.js` (where the Map may be defined and passed)
- `api/core/orchestrator.js` `#loadDocumentContext` method

## Bug 3: Stock/commodity price queries fail — "technical issue" error

**Affected queries:** Apple stock, Amazon stock, platinum price, silver price
**Working queries:** Ethereum price (CoinGecko), weather (RSS), news (RSS)

**Evidence:**
```
Roxy: I encountered a technical issue processing your request. I want to be honest: rather than provide 
potentially incorrect information, I need to acknowledge this limitation.
```

**Root cause:** The `externalLookupEngine.js` has CoinGecko for crypto prices and Google News RSS for news, but **no stock price or commodity API**. When the system can't find data, it throws an error instead of gracefully saying "I don't have access to stock prices."

**Fix:** In `api/core/intelligence/externalLookupEngine.js`, when a stock/commodity query is detected but no API is available:
1. Don't throw an error or return an error object
2. Instead return an empty result with a flag like `{ available: false, reason: 'no_stock_api' }`
3. The orchestrator should then let GPT-4 respond naturally (it knows it can say "I don't have real-time stock data")

Alternatively, this may be related to Bug 1 — the `memoryText.match` crash happens BEFORE the AI can respond, so what looks like "no stock API" might actually be the crash preventing any response at all. Fix Bug 1 first and re-test stock queries.
