Issue Description:
## Problem

Three price lookup sources are all failing in production, returning 
30-35 character responses (HTTP 429 rate limit errors):

Railway logs confirm:
- GoldAPI gold:   https://www.goldapi.io/api/XAU/USD → 32 chars
- GoldAPI silver: https://www.goldapi.io/api/XAG/USD → 30 chars  
- CoinGecko:      https://api.coingecko.com/api/v3/simple/price?ids=bitcoin,ethereum → 35 chars

All three are failing because no API key is being sent in the request 
headers. The environment variables exist in Railway but are not being 
injected into the fetch calls.

Additionally, even when prices are retrieved, the system cannot answer 
weight-based queries like "what is 50 lbs of gold worth" because there 
is no unit conversion between pounds/kilograms and troy ounces.

## Fixes Required

### 1. Fix GoldAPI key injection
The GOLDAPI_KEY environment variable exists in Railway but is not being 
sent. Find the GoldAPI fetch call in externalLookupEngine.js and ensure 
the key is sent as the x-access-token header on every request:
headers: {
'x-access-token': process.env.GOLDAPI_KEY,
'Content-Type': 'application/json'
}

### 2. Add metals.live as free no-key fallback for gold and silver
If GoldAPI fails or returns below threshold, cascade to:

Gold:   https://api.metals.live/v1/spot/gold
Silver: https://api.metals.live/v1/spot/silver

This API requires no key and no account for under 30,000 
requests/month. Do not add any authentication header — call it raw.

### 3. Replace CoinGecko with Coinbase public endpoint for crypto
Replace the CoinGecko call with Coinbase public price endpoints which 
require no key and have no rate limits for public use:

Bitcoin:  https://api.coinbase.com/v2/prices/BTC-USD/spot
Ethereum: https://api.coinbase.com/v2/prices/ETH-USD/spot

Keep CoinGecko as a secondary fallback after Coinbase in case Coinbase 
is unavailable. Do not remove CoinGecko — just move it to second 
position in the cascade.

### 4. Add unit conversion for weight-based price queries
When a user asks about a quantity of a metal (e.g. "50 lbs of gold", 
"2 kg of silver", "10 ounces of platinum"), the system must convert 
the user's unit to troy ounces before calculating value.

Conversion constants:
- 1 pound (lb)      = 14.5833 troy ounces
- 1 kilogram (kg)   = 32.1507 troy ounces
- 1 gram            = 0.0321507 troy ounces
- 1 ounce (oz avoirdupois) = 0.911458 troy ounces
- 1 troy ounce      = 1 troy ounce (no conversion needed)

The response must show the math clearly:
"50 lbs = 729.17 troy oz × $3,247/oz = $2,368,000"

Detect weight-based queries semantically — do not rely on keyword 
matching alone. The query "what would 50 pounds of gold be worth" and 
"I have 2 kilos of silver, what's the value" should both trigger this 
path.
