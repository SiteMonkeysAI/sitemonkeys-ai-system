Issue #916: [claude-fix] Retrieval Diversity Enforcement — One Dominant Memory Crowding Out All Others

Issue Description:
Problem
Semantic retrieval is broken for users with any high-scoring memory in the database. A single memory with a high relevance_score monopolizes every retrieval query regardless of topic. In observed logs, Memory 9454 appeared in 10+ boost operations and dominated all 3 recall queries (Tesla, dog name, favorite color) — returning the same memory ID for completely unrelated topics. The dog name "Max" stored moments earlier never surfaced despite being in the database.
This is a retrieval diversity failure, not a storage failure. Facts are being stored correctly but are unreachable because one high-scored memory absorbs all available result slots.
Root Cause (confirmed via code analysis)
Two compounding problems:
1. Prefilter uses relevance_score DESC instead of semantic distance
buildPrefilterQuery orders its LIMIT 500 candidate pool by relevance_score DESC — a historical importance/frequency score. This means the 500 candidates fed into cosine similarity scoring are already biased toward whichever memory has accumulated the most access history. A memory that was relevant weeks ago keeps winning the prefilter forever.
2. Zero diversity enforcement in the scoring/return pipeline
After scoring, there is no MMR (Maximal Marginal Relevance), no diversity penalty, and no deduplication. Nothing prevents the same memory ID from consuming multiple result slots or dominating across consecutive queries on different topics. defaultTopK = 10 slots are available but one memory can legally fill all 10.
Required Fix
The agent must explore the full retrieval pipeline (semantic-retrieval.js, buildPrefilterQuery, and the scoring/ranking section) and implement all three of the following. Do not implement only one — all three are required to solve this properly.
Fix 1 — Semantic prefilter ordering
When a query embedding is available, buildPrefilterQuery must use pgvector <=> distance ordering instead of relevance_score DESC. This ensures the 500-candidate pool contains the memories most semantically similar to the current query, not the historically most-accessed memories. relevance_score may remain as a tiebreaker or secondary sort but must not be the primary sort when an embedding is present.
Fix 2 — Per-query memory ID deduplication
A single memory ID must not appear more than once in the final returned results for any query. After scoring and before return, deduplicate by memory_id keeping only the highest-scoring instance of each.
Fix 3 — MMR-style diversity enforcement
After initial scoring, apply a diversity pass before finalizing results. The approach: score all candidates normally, select the top result, then for each subsequent slot apply a penalty to any candidate whose embedding is highly similar to an already-selected result. The penalty should be tunable via a constant (e.g., DIVERSITY_PENALTY_WEIGHT = 0.3, DIVERSITY_SIMILARITY_THRESHOLD = 0.85). This ensures the result set spans different topics/facts rather than clustering around one dominant memory.
Constants to add to RETRIEVAL_CONFIG
jsdiversityEnforcement: true,          // Master switch
diversityPenaltyWeight: 0.3,         // How aggressively to penalize similar results
diversitySimilarityThreshold: 0.85,  // Cosine similarity above which penalty applies
prefilterUseEmbedding: true,         // Use pgvector <=> when embedding available
