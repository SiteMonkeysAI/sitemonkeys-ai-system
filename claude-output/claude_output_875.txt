## Issue #875 Fix Summary

### Root Cause Analysis

**Issue 1 — Confidence percentage not displaying:**
The `!response.toLowerCase().includes("confidence")` check in `eli_framework.js` line 375 was too broad. The system prompt's uncertainty template instructs GPT-4 to include `(Confidence: 0.X)` in responses, and GPT-4 often naturally uses phrases like "I'm confident that...". This caused the check to silently skip the structured `🎯 **Confidence Assessment:**` block for almost every non-trivial response.

**Issue 2 — Current events queries not triggering external lookup:**
The query "Can you give me a breakdown of Apple's recent product announcements" was classified as AMBIGUOUS (Stage 1) → SEMI_STABLE (Stage 2). However, the orchestrator's `shouldLookup` decision required EITHER `matchesNewsPattern` (which only matched words like "news", "today", "breaking") OR `hasExplicitFreshnessMarker` (set only by Stage 1's `FRESHNESS_OVERRIDE_PATTERNS`). The `FRESHNESS_OVERRIDE_PATTERNS` only matched "recent information/news/updates/developments/events" — not "recent product announcements". So `shouldLookup = false` and no external lookup fired.

**Issue 3 — Hallucinated memory context:**
GPT-4 was fabricating "previous conversation" references (battery throttling lawsuits, Warren Buffett stock sales) based on its training data knowledge about Apple when no actual memory context existed for those topics. Two gaps enabled this: (1) the system prompt lacked an explicit prohibition against fabricating memory references, and (2) `FALSE_CONTINUITY_PATTERNS` in `responseContractGate.js` didn't catch phrases like "things we discussed previously" or "in our previous conversation".

---

### Changes Made

**1. `api/core/personalities/eli_framework.js`** — Fix confidence percentage display
- Changed the check from `!response.toLowerCase().includes("confidence")` to `!alreadyHasConfidencePercentage`
- `alreadyHasConfidencePercentage` specifically checks for the doctrineEnforcer's structured percentage note (`My confidence in this response is \d+%`) OR the existing 🎯 block
- This allows the confidence block to appear even when the AI response naturally uses the word "confidence"

**2. `api/core/intelligence/truthTypeDetector.js`** — Fix external lookup trigger for recent announcements
- Added two principle-based patterns to `FRESHNESS_OVERRIDE_PATTERNS`:
  - `/\brecent\b.{0,50}\b(announcements?|releases?|launches?)\b/i` — catches "recent product announcements"
  - `/\b(announcements?|releases?|launches?).{0,30}\b(recent|latest|newest|new)\b/i` — catches reverse order
- These patterns are entity-agnostic (no hardcoded company names) and work for any company/product
- When matched, Stage 1 returns SEMI_STABLE with `patterns_matched: [{ pattern: 'explicit_freshness_marker' }]`, which triggers `shouldLookup = true` in the orchestrator

**3. `api/core/orchestrator.js`** — Two changes:
- Updated `NEWS_TRIGGER_PATTERNS` to add `announcement|announced|recently announced` as a safety-net trigger for SEMI_STABLE queries about announcements
- Added a strong explicit prohibition to the system prompt: "CRITICAL — MEMORY FABRICATION IS A CATEGORY 1 TRUST VIOLATION" with explicit instruction to never claim to have discussed topics not present in the MEMORY CONTEXT section

**4. `api/core/intelligence/responseContractGate.js`** — Extend false continuity stripping
- Added 6 new patterns to `FALSE_CONTINUITY_PATTERNS` to catch additional fabricated memory reference forms:
  - `things we discussed previously`
  - `we previously discussed` / `I previously mentioned`
  - `in our previous conversation/session/chat`
  - `previously in our conversation`
  - `as we've discussed/covered`
  - `from our previous conversation/discussion`
