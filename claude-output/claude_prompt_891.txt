## FAILURE 1: MEM-007 — Critical facts not prioritized over preferences

### What the test does
1. Stores: "I'm allergic to peanuts" (safety-critical health fact)
2. Stores: "I love vanilla ice cream" (casual preference)
3. Asks: "What should you know if you're serving me food?"
4. Expects: Response LEADS with or clearly PRIORITIZES the allergy over the preference

### What actually happens
The system retrieves the ice cream preference but NOT the allergy, or treats them as equal weight. Latest response: "when it comes to ice cream, you prefer vanilla over chocolate" — allergy not mentioned at all.

### Root cause analysis
This is NOT a storage problem — both facts get stored. This is a **retrieval ranking** problem. When the query "What should you know if serving me food?" runs semantic retrieval, the allergy memory and the preference memory compete for retrieval slots. The importance scoring system (Innovation #7 per the Bible) should score health/safety facts HIGHER than casual preferences, but either:

1. The importance score assigned at storage time doesn't distinguish "allergic to peanuts" (safety-critical) from "love vanilla ice cream" (preference), OR
2. The retrieval ranking doesn't use importance scores to boost safety-critical memories, OR  
3. Both memories are stored but semantic similarity to "serving me food" happens to match "ice cream" better than "allergy" — and there's no importance-based tiebreaker

### What needs to happen
The importance scoring (calculated in `intelligent-storage.js` or wherever the semantic analyzer runs) needs to recognize health/allergy/safety facts as HIGH importance (0.8-1.0 range). The category system already has categories — there should be a `health_safety` or `medical_allergy` category that gets a high base score.

Then during retrieval, importance scores need to actually influence ranking. If they already do, verify the allergy is getting a meaningfully higher importance score than the ice cream preference. If both get 0.5, that's the bug.

### Per the Bible (Innovation #7 - Memory Importance Scoring):
> "Importance scores inform all system decisions - what to retrieve first, what to display most prominently, what to protect from overwrite."

The allergy MUST score higher than the preference. This is the system working "as it should be."

## FAILURE 2: CMP2 — International names not preserved through compression

### What the test does
1. Stores a message containing three names: Zhang Wei, Björn Eriksson, José García
2. Asks: "Who are my contacts?"
3. Expects: All three names appear in the response with correct spelling

### What actually happens
Response says "you've mentioned two individuals who could be considered your contacts" — names are gone. The compression pipeline strips or mangles international names with special characters (ö, é, í) and non-English name patterns.

### Root cause analysis
The compression prompt sent to GPT-4o-mini likely doesn't explicitly instruct preservation of proper nouns. When compressing "My contacts are Zhang Wei from Shanghai, Björn Eriksson from Stockholm, and José García from Madrid," GPT-4o-mini may:
- Drop names entirely as "unnecessary detail"
- Mangle special characters (ö → o, é → e, or worse)
- Summarize to "three international contacts" losing all specifics

Note: The previous PR already added CMP2 international name re-injection logic (line ~1721-1729 in `intelligent-storage.js`). This code may not be working, or the names may be getting lost BEFORE that re-injection step runs. Investigate whether the re-injection is actually firing and whether the names exist in the input to that step.

### What needs to happen
1. The compression prompt MUST explicitly instruct: "Preserve all proper nouns (names of people, places, companies) exactly as written, including accented characters and non-English formatting."
2. If the CMP2 re-injection code exists, verify it's actually executing by adding a log line. If it IS executing but names are still lost, the regex or matching logic in the re-injection is wrong.
3. After compression, verify the output contains all input proper nouns. If any are missing, re-inject them.

## FAILURE 3: EDG3 — Pricing details from long input not preserved

### What the test does
1. Sends a long business description containing specific pricing tiers (dollar amounts, package names, percentages)
2. Asks about the pricing details
3. Expects: Specific pricing numbers recalled

### What actually happens
Response shows generic business context ("TechFlow Inc.") but no specific pricing numbers. The compression is stripping numeric details from long inputs.

### Root cause analysis
Long inputs hit compression harder. A 200+ word business description with pricing tiers like "$499/month Basic, $999/month Pro, $2,499/month Enterprise" gets compressed and the specific dollar amounts are treated as expendable detail. The compression prompt doesn't know that pricing/financial data is high-value content that must survive.

The existing `aggressivePostProcessing` function is supposed to protect pricing lines, but it may not be matching the specific format the test uses, or the protection happens after content is already lost.

### What needs to happen
1. The compression prompt needs explicit instruction: "Preserve all specific numbers, dollar amounts, percentages, dates, and quantitative data exactly as stated."
2. For long inputs (>100 tokens), consider a two-pass approach: first extract key facts/numbers, then compress the narrative around them while keeping the extracted data intact.
3. Verify the `aggressivePostProcessing` pricing protection regex actually matches the test input format.

### Root cause analysis

The test's pass/fail logic probably checks for certain keywords or patterns and the current response wording doesn't match what the test expects. The test may be looking for stronger language like "does not exist" or "is fictional" and the system's more diplomatic phrasing ("there isn't a known historical event called...") triggers a false failure.

### What needs to happen
**FIRST:** Examine the test code for TRUTH-017. Look at what regex/keyword check determines pass vs fail. The response "there isn't a known historical event" is objectively correct — the system is NOT hallucinating. If the test is failing a correct response, the test validation logic needs adjustment.

**IMPORTANT EXCEPTION TO THE "DON'T MODIFY TESTS" RULE:** If investigation confirms the test is producing a false failure on a correct system response, then the test validation logic (not the test scenario) should be adjusted to correctly recognize the system's response as a pass. This is fixing a test BUG, not rewriting a test to match broken behavior. Document exactly what was wrong with the test check and why the fix is appropriate.
