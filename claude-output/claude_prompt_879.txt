Issue 2: Cross-topic memory contamination — Apple memories bleeding into Tesla queries at retrieval level
Title: [claude-fix] Entity-boosted Apple memories injected into Tesla queries — cross-topic contamination not prevented at retrieval stage

Problem
When a user asks "What have we discussed about Tesla?", the system retrieves and injects Apple-tagged memories (IDs 9047, 9045, 8717, 8614, 8436) because they are entity-boosted for "Apple" and have high similarity scores. GPT-4 then presents these Apple facts as prior conversation history about Tesla.
Actual response produced:

"our past discussions have mostly revolved around Apple, its battery throttling issue, and how the actions of individuals like Warren Buffett can affect stock prices"

This is wrong on two levels:
Level 1 — Retrieval: The memory retrieval pipeline is injecting memories tagged to entity "Apple" in response to a query about entity "Tesla". Entity boosting is working correctly to surface Apple memories for Apple queries, but there is no corresponding guard to prevent Apple memories from being injected into non-Apple queries when semantic similarity happens to pull them in.
Level 2 — Response generation: The FALSE_CONTINUITY_PATTERNS in responseContractGate.js did not catch the phrase "our past discussions have mostly revolved around" — this pattern was not in the list added by PR #877.
Both layers must be fixed.
Fix 1 — Retrieval layer (primary fix)
In the memory retrieval pipeline, when a query contains a named entity (Tesla, Apple, a person's name, a company, etc.), memories that are tagged to a different named entity should not be injected unless the query explicitly asks for cross-topic connections. The entity extracted from the query should act as a filter or strong de-boost signal against memories carrying a conflicting entity tag.
If the query entity is "Tesla" and a retrieved memory's entity tags contain only "Apple" (not Tesla), that memory's score should be heavily penalized or excluded before injection. Memories with no entity tag, or with matching entity tags, pass through normally.
Fix 2 — Pattern layer (secondary fix)
Add the following patterns to FALSE_CONTINUITY_PATTERNS in responseContractGate.js:

our past discussions have mostly revolved around
our discussions have revolved around
our conversations have mostly been about
we've mostly talked about
most of our discussion has been about

Expected behavior
When user asks about Tesla and no Tesla memories exist, the system responds: "I don't have any stored context about Tesla in our previous conversations. Would you like to discuss Tesla now?" It does not volunteer Apple memories, financial memories, or any other off-topic memories as a substitute.
