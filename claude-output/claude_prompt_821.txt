Issue #821: claude-fix [P0] Fix context Temporal Dead Zone crash in orchestrator.js processRequest

Issue Description:
Problem
Document uploads crash the system with:
[ORCHESTRATOR ERROR] Request failed: Cannot access 'context' before initialization
ReferenceError: Cannot access 'context' before initialization
    at Orchestrator.processRequest (file:///app/api/core/orchestrator.js:1072:34)
This is a JavaScript Temporal Dead Zone (TDZ) bug — a const or let variable is referenced before its declaration executes.

Root Cause (Confirmed)
In processRequest, there is a document injection gating block that runs before the context variable is declared:
javascript// THIS RUNS EARLY (around line ~1072):
let effectiveDocumentData = documentData;
if (documentData) {
  const refersToDocument = /\b(document|file|pdf|upload|summary|summarize|contents|attachment|that file|the file|this file|what I uploaded|I just loaded|I just uploaded)\b/i.test(message);
  const isDocumentReview = context?.truthType === 'DOCUMENT_REVIEW' || earlyClassification?.classification === 'document_review';
  //                        ^^^^^^^ THIS IS THE BUG
  if (!refersToDocument && !isDocumentReview) {
    this.log('[DOCUMENTS] ⏭️ Skipping document injection — query does not reference document');
    effectiveDocumentData = null;
  }
}

// THIS RUNS LATER:
const context = this.#assembleContext(...)
context?.truthType is accessed before const context = this.#assembleContext(...) executes. Even with optional chaining (?.), JavaScript throws a ReferenceError because context is in the temporal dead zone.

Why It Only Crashes On Document Upload
The bug is inside if (documentData). Normal chat without document attachments skips this block entirely, so the TDZ is never hit. The moment a user uploads a file, documentData is truthy, this block executes, and the crash occurs.

The Fix
Replace the line that references context with information already available at that point in execution:
Find this line:
javascriptconst isDocumentReview = context?.truthType === 'DOCUMENT_REVIEW' || earlyClassification?.classification === 'document_review';
Replace with:
javascriptconst cls = earlyClassification?.classification;
const isDocumentReviewByClassifier = cls === 'document_review' || cls === 'DOCUMENT_REVIEW';
const hasDocVerb = /\b(summarize|summary|review|analyze|explain)\b/i.test(message);
const isDocumentReview = isDocumentReviewByClassifier || (refersToDocument && hasDocVerb);
This removes the illegal context reference entirely. Key design decisions:

earlyClassification is already available at this point in the function — no reordering needed.
Classifier match is case-insensitive — handles both 'document_review' and 'DOCUMENT_REVIEW' since the codebase uses both conventions.
Action verbs only count when the user also references the document — refersToDocument && hasDocVerb prevents "Can you explain what Ethereum is?" from being treated as document review when a previous upload left documentData populated. Without this guard, any query containing "explain" or "analyze" would keep stale document context injected.

Do NOT:

Reorder the function to move context declaration earlier (risks side effects)
Add let context at the top with a placeholder value (changes semantics)
Wrap in try/catch (masks the bug instead of fixing it)
