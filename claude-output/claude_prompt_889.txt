<h2>Root Cause</h2>
<p>The compression pipeline (likely GPT-4o-mini based) is compressing personal facts down to 1-2 tokens. At this level:</p>
<ol>
<li>The embedding model can't generate a semantically meaningful vector</li>
<li>Even if retrieved, GPT-4 can't use a single word like "salary." to construct a useful response</li>
<li>The memory is wasting a database row and embedding computation for zero value</li>
</ol>
<h2>Required Fix</h2>
<h3>Principle</h3>
<p>Compression must preserve enough semantic content for embeddings to function AND for GPT-4 to use the memory meaningfully. A memory should contain: the subject, the fact, and enough context to be self-explanatory.</p>
<h3>Specific Requirement</h3>
<p><strong>Enforce a minimum token floor of 8-10 tokens for any stored memory.</strong></p>
<p>Implementation approach (explore the codebase to find the best location):</p>
<ol>
<li>After compression runs, check the token count of the output</li>
<li>If below the floor (8-10 tokens), either:
<ul>
<li>Re-run compression with a prompt that explicitly requires preserving more detail, OR</li>
<li>Use the original uncompressed content if it's already short enough (under ~50 tokens), OR</li>
<li>Apply a lighter compression pass that preserves subject + predicate + key details</li>
</ul>
</li>
<li>The goal: every stored memory should be a readable, self-contained fact like <code>"User salary: $95,000/year at current job"</code> not <code>"salary."</code></li>
</ol>
