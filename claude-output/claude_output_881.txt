## Issue #881 Fix: Semantic Detection for Conversational Current-Event Queries

### Problem
External lookup was not triggering for conversational queries about named entities' recent actions because the detection relied on explicit freshness keywords ("current", "latest", "today") and hardcoded geopolitical terms. Queries like "Did Saudi Arabia make a big commitment" and "What is Schumer demanding from Trump" bypassed all lookup triggers.

### Root Causes
1. "Did [entity] [action]" queries → returned AMBIGUOUS→SEMI_STABLE but `matchesNewsPattern` used hardcoded terms ("venezuela", "ukraine", etc.) so Saudi Arabia/Coast Guard didn't match → `shouldLookup = false`
2. "What is [entity] [action gerund]" queries → matched PERMANENT pattern due to "what is" → misclassified as definitional, bypassing current-event lookup path
3. "Seems like [entity] has something going on" → no freshness markers → AMBIGUOUS→SEMI_STABLE → same dead end
4. Follow-up queries lost volatile context from prior turn

### Changes

**`api/core/intelligence/truthTypeDetector.js`**
- Added `hasNamedEntityActionPattern(query)` helper function with local proper-noun detector (can't import from externalLookupEngine due to circular dependency)
- Added detection block in `detectByPattern` AFTER conversational/personal check, BEFORE PERMANENT patterns
- 4 structural patterns: "did [entity] [action]", "[entity] has something going on", "seems like [entity]...", "what is [entity] [action gerund]"
- Returns SEMI_STABLE with pattern `entity_action_current_event` — prevents PERMANENT misclassification

**`api/core/intelligence/externalLookupEngine.js`**
- Added `isCurrentEventQuery(query)` exported function using `hasProperNouns()` + same 4 structural patterns
- Updated `isLookupRequired` to call `isCurrentEventQuery` and add `current_event_query_named_entity` reason with `priority = 'high'` — ensures `lookup()` internal gate passes even if orchestrator somehow skips `shouldLookup`
- Updated `selectSourcesForQuery` to route `isCurrentEventQuery` matches to NEWS sources directly (not just DuckDuckGo fallback)
- Added `isCurrentEventQuery` to default export

**`api/core/orchestrator.js`**
- Updated import to include `isCurrentEventQuery` and `hasProperNouns` from externalLookupEngine
- Added `isSemanticCurrentEventQuery = isCurrentEventQuery(message)` check
- Added `isVolatileFollowUp` check: detects follow-up queries where prior user turns had named-entity current-event queries, inheriting volatile classification
- Both added to `shouldLookup` conditions
- Both added to debug logging

### Design Principle
The fix uses **structural semantic classification** (proper noun + action verb structure) rather than keyword expansion. This catches any query about what a named public figure, country, or organization is currently doing — regardless of how the question is phrased — without hardcoding entity names.
