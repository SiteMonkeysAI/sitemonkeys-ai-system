## FAILURE 3 [P0]: Document Context Is "Sticky" — Bleeds Into ALL Subsequent Unrelated Queries

After the .docx loaded successfully, EVERY subsequent query received 807 tokens of forensic report content — including stock price queries that have zero relevance to the document.

**Walmart stock query (asking about stock price, NOT about the document):**

**Apple stock query (same — stock price, not document):**
```

```

A forensic report about iPhone unauthorized access was injected into both stock price queries. This causes:
- **807 wasted tokens per query** for the rest of the session
- **Wrong storage tagging** — Walmart stock response got `[STORE] Tagged memory with [SOURCE:document]` because document context was present in the request
- **Business boilerplate triggers** — document presence may be confusing the classification layer
- **"DOCUMENT CONTEXT" anchor validation errors** — `[ANCHOR-VALIDATOR] ⚠️ Rejected invalid unicode anchor: DOCUMENT CONTEXT` appears on multiple queries because document context markers are leaking into anchor processing

**Expected behavior:** Document context should only be injected when the query is about the document (DOCUMENT_REVIEW classification, or user explicitly references the uploaded content). Market queries, news queries, and general questions should NOT receive document injection. The orchestrator should gate document injection on query intent.

## SYSTEMIC ROOT CAUSES (from deep log analysis)

These are the underlying patterns causing multiple failures above. Fixing these prevents recurrence.

### ROOT CAUSE A [P1]: User Queries Stored Verbatim as Memory (Not Extracted Facts)

**Observed:** Multiple memory entries contain the user's previous questions stored word-for-word instead of extracted facts:
- Memory 8611: `"What's the most up to date information from the news Greenland"` — this is a user query, not a fact
- Memory 8612: `"What's the most up-to-date information on Columbia since we went in and took their leader"` — same pattern

**Impact:** Retrieval degenerates into "question matches question" with similarity ~1.0, consuming token budget while providing zero informational value. Memory 8617 achieved similarity=1.000 on the Ethereum query — extremely likely to be a stored previous crypto question.

**Likely cause:** The automatic storage path that runs when `detectExplicitMemoryRequest` returns `false` is capturing user prompts instead of extracting facts from the assistant's response.

**Investigation target:** The automatic store path in `intelligent-storage.js` that fires after `detectExplicitMemoryRequest` returns `{isExplicit: false}`. Trace what content it receives and stores. It should be extracting facts from the AI response, not storing the user's question.

**To confirm:** Pull the actual stored text of memory IDs 8611, 8612, and 8617 from the database. If they contain question phrasing instead of factual content, this is confirmed.

### ROOT CAUSE B [P1]: Entity Boost Matching Common English Words

**Observed:** The entity extraction treats common English words as named entities, causing massive false-positive boosts:
- **"No"** → matched 30+ memories with +2.0 priority boost each
- **"With"** → matched 15+ memories
- **"By"** → matched 10+ memories
- **"All", "Hit", "File", "Content", "Control", "Access"** → all treated as entities

**Log evidence (sample of 30+):**
```
[ENTITY-BOOST] Memory 2914: Contains entities [No]
[ENTITY-BOOST] Memory 692: Contains entities [No]
[ENTITY-BOOST] Memory 4011: Contains entities [No]
[ENTITY-BOOST] Memory 2965: Contains entities [No]
[ENTITY-BOOST] Memory 8425: Contains entities [No]
```

**Impact:** 70+ false positive entity boosts on the forensic document query. Every memory containing "No" got a +2.0 tier boost, massively polluting retrieval ranking.

**Likely cause:** Naive "capitalized token = entity" extraction, or sentence-start capitalization being treated as a proper noun.

**Fix target:** Add stopword filtering + sentence-start capitalization guard in the entity extractor (wherever `ENTITY-BOOST` pulls its entity list from). Common English words should never be treated as named entities regardless of capitalization.

### ROOT CAUSE C [P2]: "No extractable facts from." Stored as Memory

```
[STR1-DEBUG]   Memory 1: "No extractable facts from."
```

This appears in multiple queries. The intelligent storage system tried to extract facts from something, found nothing, and stored the failure message itself as a memory entry. This pollutes future retrievals.

**Fix:** The storage system should NOT store extraction failure messages as memory entries. If no facts are extracted, store nothing.

### ROOT CAUSE D [P2]: Contradictory Storage Tags

On the Walmart stock query:
```
[STORE] Tagged memory with [SOURCE:document]
[STORE] Reason: External real-time data (prices/news) should not be stored as persistent memory
```

The response is tagged as document-sourced AND skipped as external data simultaneously. The document tag fires because document context (forensic report) was present in the request — this is a downstream consequence of Failure 3 (document bleed). Fixing Failure 3 likely fixes this.

### ROOT CAUSE E [P2]: ANCHOR-VALIDATOR "DOCUMENT CONTEXT" Errors

```
[ANCHOR-VALIDATOR] ⚠️ Rejected invalid unicode anchor: DOCUMENT CONTEXT
```

Document context section markers are leaking into anchor processing. Another downstream consequence of document context being present where it shouldn't be.

### ROOT CAUSE F [P3]: Memory 2903 — 1,322 Token Single Memory Entry

Memory 2903 contains an entire document summary (Site Monkeys .docx content) as a single 1,322-token entry. When retrieved, it consumes nearly the entire memory budget:

```
[RETRIEVAL-GROUPING] ✅ Added high-priority memory 2903 (1322 tokens)
[RETRIEVAL-GROUPING] ⚠️  High-priority memory 2824 would exceed budget + overflow
```

One memory entry blocks all other memories from being retrieved. The storage system should chunk or summarize large content before storing, with a per-entry token cap.

### ROOT CAUSE G [P3]: Performance — 10-17 Second Response Times

- Query 1 (Apple event): Total 15,041ms
- Query 5 (Colombia): Total 10,367ms
- Query 8 (document summary): AI generation took 17,186ms

Memory retrieval is fast (230-989ms). The bottleneck is AI generation. For context: the document summary query injected 1,470 tokens of memory + 807 tokens of document context + the system prompt — the prompt is likely exceeding optimal size for GPT-4 response time.
