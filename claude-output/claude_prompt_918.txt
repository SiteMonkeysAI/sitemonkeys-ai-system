## PART 2: Real Improvements — Compression Intelligence

These are persistent failures that have never been properly fixed because previous PRs tried prompt instructions instead of pipeline improvements.

### CMP2: International Names Not Surviving Compression

**The problem:** When the user says "My contacts are Zhang Wei, Björn Eriksson, and José García," the compression pipeline (GPT-4o-mini) drops the names. Retrieval then has nothing useful to return.

**What's been tried:** PR #891 updated compression prompt Rule 3 to say "PRESERVE ALL PROPER NAMES EXACTLY AS WRITTEN" with examples. The names are STILL being lost.

**What needs to happen — investigate the actual compression pipeline:**

1. Add diagnostic logging to the compression step. BEFORE compression, log the input. AFTER compression, log the output. Compare them for name preservation. This will show exactly where names are being lost.

2. Check if the CMP2 name re-injection code (added in a previous PR around line 1721-1729 in `intelligent-storage.js`) is actually executing. Add a log line to confirm. If it's executing but names are still missing, the regex matching is wrong.

3. The fundamental question: Is GPT-4o-mini ignoring the compression prompt instructions about names? If it is, prompt instructions alone won't fix this. Consider a post-compression validation step:
   - Extract proper nouns from the original input (regex for capitalized multi-word sequences, names with diacritics)
   - After compression, check if those nouns exist in the output
   - If any are missing, append them to the compressed output
   - This is a pipeline fix — it mechanically guarantees name preservation regardless of what GPT-4o-mini decides to do

4. The token floor fix means compressed memories are at least 8 tokens. But if those 8 tokens are "Has three contacts." instead of "Contacts: Zhang Wei, Björn Eriksson, José García" — the floor doesn't help. The compression output needs to be evaluated for CONTENT quality, not just length.

### EDG3: Pricing Details Not Surviving Long-Input Compression

**The problem:** User provides a detailed business description with specific pricing ($499/month Basic, $999/month Pro, etc.). After compression, the specific dollar amounts are gone.

**What needs to happen:**

1. Same diagnostic approach — log before and after compression for long inputs. See what's being lost.

2. The `aggressivePostProcessing` function supposedly protects pricing lines. Check if it's actually matching the format used in the test. If the test says "$499/month" and the regex only matches "$499.00" — that's why it fails.

3. For long inputs (>100 tokens), consider a pre-extraction approach:
   - Before compression, extract key numeric data (dollar amounts, percentages, dates)
   - Pass the narrative to compression normally
   - After compression, verify the numeric data survived
   - If not, append the extracted numbers to the compressed output
   - This guarantees quantitative data preservation mechanically

4. The Bible says compression should achieve "40-60% reduction without sacrificing usefulness." If a business description gets compressed so aggressively that pricing disappears, that's broken compression.

### General Compression Principle

Both CMP2 and EDG3 point to the same root: the compression pipeline treats all content as equally compressible. Names and numbers are NOT compressible — they're the high-value content that must survive. The narrative around them is what should compress.

**"Compress the narrative, preserve the data."** Names, numbers, dates, codes, identifiers — these are facts. The sentences connecting them are narrative. Compress the narrative, keep the facts verbatim.
