Issue #932: [Security] Add rate limiting to POST /api/chat — currently unlimited, resource exhaustion vector

Issue Description:
## Problem
 
`POST /api/chat` has no rate limiting. It is the most expensive
endpoint in the system — every request triggers OpenAI API calls,
memory retrieval, and pgvector queries. An unlimited endpoint
means a single IP can flood the system with requests, running up
API costs and degrading performance for all users.
 
The audit also found that server.js has a 50MB body size limit
on this endpoint with no request throttling, compounding the risk.
 
`express-rate-limit` is already installed and used elsewhere in
the codebase (migration routes), so this requires no new dependency.
 
## Required Fix
 
Add a rate limiter specifically for the chat endpoint in server.js.
Place it BEFORE the route handler, not as global middleware.
 
  import rateLimit from 'express-rate-limit';
 
  const chatRateLimit = rateLimit({
    windowMs: 60 * 1000,        // 1 minute window
    max: 20,                     // 20 requests per minute per IP
    standardHeaders: true,
    legacyHeaders: false,
    message: {
      error: 'Too many requests. Please wait a moment before sending
      another message.'
    }
  });
 
  app.post('/api/chat', chatRateLimit, chatHandler);
 
20 requests per minute is generous for legitimate use (one message
every 3 seconds) while blocking automated flooding.
 
## What NOT to Touch
 
- Do not add global rate limiting to all routes
- Do not modify the chat handler logic itself
- Do not change any other route's rate limiting
- Do not change the body size limit (separate concern)
- Only add the rate limiter to POST /api/chat
 
## Stop and Report If
 
- The chat route is registered in a way that makes inserting
  middleware before it complex or risky
- express-rate-limit version in package.json is incompatible
  with this syntax
- Any existing middleware conflicts with rate limiting
 
## Validation
 
After deploy:
- Confirm POST /api/chat responds normally for a single request
- Confirm the rate limiter is present in server.js before the
  chat route handler
- Normal conversation use should be completely unaffected

TITLE:
[Security] Restrict file upload to safe types only — currently
accepts all file types with no filtering
 
LABELS: security, high, hardening
 
BODY:
 
## Problem
 
`api/upload-file.js` lines 17-20 configure multer with:
 
  fileFilter: cb(null, true)
 
This accepts every file type with no restriction. An attacker can
upload executable files, scripts, or malicious content to the
server. Even if files are not executed, storing arbitrary file
types creates risk.
 
The system's legitimate use case for uploads is document analysis
(PDFs, Word docs, text files). Only those types should be accepted.
 
## Required Fix
 
Replace the permissive fileFilter with an explicit allowlist:
 
  const ALLOWED_MIME_TYPES = [
    'application/pdf',
    'application/msword',
    'application/vnd.openxmlformats-officedocument
      .wordprocessingml.document',
    'text/plain',
    'text/csv',
    'application/json'
  ];
 
  const ALLOWED_EXTENSIONS = [
    '.pdf', '.doc', '.docx', '.txt', '.csv', '.json'
  ];
 
  fileFilter: (req, file, cb) => {
    const ext = path.extname(file.originalname).toLowerCase();
    if (
      ALLOWED_MIME_TYPES.includes(file.mimetype) &&
      ALLOWED_EXTENSIONS.includes(ext)
    ) {
      cb(null, true);
    } else {
      cb(new Error('File type not allowed. Accepted types: PDF,
        Word documents, text files, CSV, JSON'), false);
    }
  }
 
Also add a file size limit if one is not already present:
 
  limits: { fileSize: 10 * 1024 * 1024 }  // 10MB max
