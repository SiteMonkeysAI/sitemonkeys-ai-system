# [claude-fix] CMP2 failure is retrieval, not compression — stored "Contacts:" memory not found by "Who are my contacts?" query

## The Evidence

Compression diagnostic logs from 2026-03-12 prove the names survive the entire compression pipeline:

```
INPUT: "My contacts are: Dr. Xiaoying Zhang-Müller, Björn O'Shaughnessy, and José García-López"
RAW_OUTPUT: "Contacts: Dr. Xiaoying Zhang-Müller, Björn O'Shaughnessy, José García-López"
POST_POSTPROCESS: "Xiaoying Zhang-Müller, Björn O'Shaughnessy, José García-López"
CMP2_POST_CHECK: all 9 name(s) survived aggressivePostProcessing
```

The names are stored correctly in the database with embeddings generated. When the test later asks "Who are my contacts?" — the response says "the contacts you've mentioned are:" with NO names, or "I don't have information about your contacts."

**The compression pipeline is working. The retrieval pipeline is not finding the stored memory.**

## What Needs To Be Investigated

The query "Who are my contacts?" should match a memory containing "Contacts: Dr. Xiaoying Zhang-Müller, Björn O'Shaughnessy, José García-López" — the word "contacts" appears in both. This should work through either semantic similarity or keyword matching.

### Investigation steps:

1. **Check the semantic similarity score.** When "Who are my contacts?" runs through retrieval, what similarity score does the contacts memory get? Is it below the threshold? Log the candidate memories and their scores for this specific query.

2. **Check if keyword boost is firing.** The keyword boost system should match "contacts" in the query to "Contacts:" in the stored memory. Is the keyword extraction finding "contacts" as a keyword? Is it matching against the stored memory content?

3. **Check if the memory is even in the candidate set.** The retrieval does a pgvector similarity search. Is the contacts memory even being returned as a candidate, or is it being excluded before scoring? There may be a user_id filtering issue or the embedding similarity is so low it doesn't make the initial cut.

4. **Check the embedding quality.** The stored content after post-processing is `"Xiaoying Zhang-Müller, Björn O'Shaughnessy, José García-López"` — note that the word "Contacts:" was stripped during post-processing. If the stored memory is just a list of names without the word "Contacts", the embedding won't match well against "Who are my contacts?" This could be the root cause — aggressivePostProcessing stripped the label that makes semantic matching work.

5. **Compare with memories that DO retrieve successfully.** "Favorite color: purple" retrieves when asked "What's my favorite color?" — that works because "color" appears in both query and stored memory. "Dog: Max" retrieves when asked about pets. Check what's different about those successful retrievals versus the contacts memory.

## Most Likely Root Cause

Look at the POST_POSTPROCESS output: `"Xiaoying Zhang-Müller, Björn O'Shaughnessy, José García-López".`

The word "Contacts:" was stripped. The stored memory is just a list of names with no semantic label. The embedding of three foreign names has almost zero semantic similarity to "Who are my contacts?" And without "contacts" as a keyword in the stored text, the keyword boost can't fire either.

If this is confirmed, the fix is: **ensure aggressivePostProcessing preserves the category label** (the "Contacts:" prefix). The label is what makes the memory findable. Stripping it makes the memory invisible to retrieval.

This same pattern may explain other retrieval misses throughout the system — if post-processing strips the semantic labels that connect stored facts to natural language queries, retrieval will fail even with perfect compression.
