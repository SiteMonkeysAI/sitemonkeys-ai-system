Problem 3 (P1): Truth Type Detector Misclassifies Personal Memory Queries
Symptoms
When the user asks "Do you recall names of my monkeys?", the system:

Classifies it as SEMI_STABLE with high_stakes: true
Triggers an external lookup
Searches Google News RSS for "recall names monkeys"
Returns irrelevant news articles about actual monkeys

This is a personal memory recall query. It should use ONLY persistent memory retrieval, not external lookup.
Evidence from Production Logs
[externalLookupEngine] Lookup requested for: "Do you recall names of my monkeys..."
[externalLookupEngine] Using news fallback for volatile/semi-stable query
[externalLookupEngine] News RSS query cleaned: "Do you recall names of my monkeys..." → "recall names monkeys"
The external lookup then injects 1,253 chars of irrelevant news content AND the storage pipeline skips saving the response because it's tagged as external data:
[STORE] ⏭️ Skipping intelligent storage for volatile external data response
Where to Investigate

truthTypeDetector.js (likely in api/core/intelligence/) — Find the pattern detection logic. The query "Do you recall names of my monkeys?" contains clear personal memory signals:

"Do you recall" = memory recall request
"my" = personal/possessive
"names of my" = requesting stored personal facts

The detector should recognize these patterns and classify as PERMANENT (personal facts don't change) or skip external lookup entirely.
Add personal memory detection patterns — Before checking for VOLATILE/SEMI_STABLE/PERMANENT, the detector should check for personal recall signals:

do you recall, do you remember, what do you know about my, tell me about my
Possessive pronouns + recall verbs: my + [noun] combined with recall, remember, know
Explicit memory references: from our conversations, I told you, we discussed

When these patterns are detected, the query should be classified as PERSONAL_RECALL (or equivalent) and external lookup should be SKIPPED entirely.
externalLookupEngine.js — Find where the decision to trigger external lookup is made. Add a gate that prevents external lookup when the query is classified as personal recall. The memory gate already exists for market queries:

   [ORCHESTRATOR] [MEMORY-GATE] intent=market_query memory_injected_tokens=0 reason=external_market_data_query
A similar gate should exist for personal recall queries but in reverse — memory should be the ONLY source, and external lookup should be blocked.
Success Criteria

 "Do you recall names of my monkeys?" triggers memory retrieval ONLY, no external lookup
 "What's my favorite color?" triggers memory retrieval ONLY, no external lookup
 "Tell me about my children" triggers memory retrieval ONLY, no external lookup
 Queries that genuinely need external data (news, prices, weather) still trigger external lookup
 No Google News RSS searches for personal recall queries appear in logs

Symptoms
The memory retrieval logs show stored content that should never have been persisted:

Where to Investigate

Search for strings like:

"No relevant facts"
"No extractable facts"
"no personal facts"

Find where these strings are generated and ensure the storage pipeline treats them as "skip" signals, not content to store.
