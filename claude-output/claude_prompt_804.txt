## AUDIT AREA 1: Document Upload Pipeline (PDF files don't load)

### Symptom
```
[UPLOAD] File Prenup Chris and Myndy.pdf skipped storage: contentExtracted=false, type=document
[UPLOAD] extractedDocuments Map size after upload: 0
```

User uploads PDF → system says "I don't have any document loaded." Three different PDFs tested, all fail.

### Required Investigation
1. Open `api/upload-for-analysis.js` and trace `processFile()` completely
2. Find every code path that sets `contentExtracted = true`
3. Currently ONLY these work:
   - DOCX files (via mammoth) 
   - Text files (.txt, .md, .csv, .json — added in PR #802)
4. **PDF files are NOT handled** even though `pdf-parse` is in package.json dependencies
5. Also check: Are image files handled? (.png, .jpg for OCR or description?)

### Required Fix
Add PDF text extraction to `processFile()`:
```javascript
// BEFORE the text file handler, AFTER the DOCX handler
if (file.mimetype === 'application/pdf' || path.extname(file.originalname).toLowerCase() === '.pdf') {
    try {
        const pdfParse = (await import('pdf-parse')).default;
        const pdfData = await pdfParse(file.buffer);
        if (pdfData.text && pdfData.text.trim().length > 0) {
            docxAnalysis = { value: pdfData.text };
            contentExtracted = true;
            console.log(`[UPLOAD] PDF extracted: ${pdfData.text.length} chars, ${pdfData.numpages} pages`);
        } else {
            console.log(`[UPLOAD] PDF parsed but no text extracted (scanned/image PDF?)`);
        }
    } catch (pdfErr) {
        console.error(`[UPLOAD] PDF extraction failed:`, pdfErr.message);
    }
}
```

### Verification
After fix, this log sequence should appear:
```
[UPLOAD] PDF extracted: XXXX chars, X pages
[UPLOAD] Stored document for key: X, content length: XXXX
[UPLOAD] extractedDocuments Map size after upload: 1
...
[DOC-LOAD] Looking up document. Map size: 1, Map keys: [X]
[HANDOFF:DOCUMENT-LOAD→CONTEXT] ✅ Document found
```

## AUDIT AREA 6: External Lookup Engine Sending Full User Queries to Google News

### Symptom
```
[externalLookupEngine] Fetching from Google News RSS (https://news.google.com/rss/search?q=Can%20you%20summarize%20what's%20in%20that%20document%20I%20just%20loaded&hl=en-US&gl=US&ceid=US:en)
```

The system is searching Google News for "Can you summarize what's in that document I just loaded" — this is the raw user query, not an extracted search term. This wastes external API calls and returns irrelevant results (Anthropic articles, Android phone articles, etc.).

### Required Investigation
1. Open `api/core/intelligence/externalLookupEngine.js`
2. Find the truth type detection that classifies "Can you summarize that document" as SEMI_STABLE/AMBIGUOUS
3. Find where the search query is constructed for Google News RSS
4. Check: Is there any query extraction/reformulation before sending to external sources?
5. The truth type detector is misclassifying document queries as needing external lookup

### Required Fix
Two fixes needed:
1. **Truth type detection**: Document-related queries ("summarize this document", "explain that document", "what's in the file I uploaded") should NOT trigger external lookup. These should be classified as INTERNAL (document context only).
2. **Query reformulation**: When external lookup IS needed, extract the actual search topic from the query. "What is the current price of silver" should search "silver price today", not the full sentence. Consider adding a simple extraction:
   ```javascript
   // Strip conversational fluff from search queries
   function extractSearchQuery(userQuery) {
       // Remove common preambles
       let q = userQuery.replace(/^(can you |what is |what's |how much |tell me )/i, '');
       // Keep the substantive part
       return q.trim();
   }
   ```

### Verification
- Document queries should NOT trigger external lookup at all
- External searches should use extracted keywords, not raw user queries
