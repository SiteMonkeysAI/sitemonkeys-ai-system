---SUMMARY---
**File modified:** `server.js` (lines 218–237)

**What changed:**

Replaced the insecure `origin: true` CORS configuration with an explicit origin allowlist and a callback-based origin validator.

**Before:**
```js
app.use(
  cors({
    origin: true,       // reflects any origin — dangerous with credentials
    credentials: true,
  }),
);
```

**After:**
```js
const allowedOrigins = [
  process.env.PRODUCTION_URL,
  ...(process.env.NODE_ENV !== "production"
    ? ["http://localhost:3000", "http://localhost:3001"]
    : []),
].filter(Boolean);

app.use(
  cors({
    origin: (origin, callback) => {
      if (!origin || allowedOrigins.includes(origin)) {
        callback(null, true);
      } else {
        callback(new Error("Not allowed by CORS"));
      }
    },
    credentials: true,
  }),
);
```

**Why this is safe:**
- `!origin` passes requests with no Origin header (server-to-server, curl, Postman) — standard practice.
- `allowedOrigins.includes(origin)` only permits explicitly listed origins.
- `process.env.PRODUCTION_URL` must be set in Railway environment variables for the production frontend to work — this is a required deploy step.
- Localhost variants are only included when `NODE_ENV !== "production"`, so they are never present in production.
- `.filter(Boolean)` prevents a `null`/`undefined` `PRODUCTION_URL` from accidentally matching missing-origin requests.
- No other server configuration, middleware order, routes, or session handling was touched.

**Required follow-up action:** Ensure `PRODUCTION_URL` is set as a Railway environment variable (e.g., `https://your-app.up.railway.app`) before deploying, otherwise the production frontend will be blocked by CORS.
