---SUMMARY---

## Root Causes Confirmed

**Root Cause #1 CONFIRMED**: `hierarchyRouter.js` defaulted AMBIGUOUS claim type to `EXTERNAL_FIRST` in non-`site_monkeys` modes. Queries like "Hello", "I have a dog named Max", "Remember my favorite color is purple" matched no business/factual patterns → AMBIGUOUS → EXTERNAL_FIRST → external lookup triggered → `hasExternal=true` → storage skipped.

**Root Cause #2 CONFIRMED**: `server.js` storage gate used a binary `result.sources?.hasExternal` check that skipped ALL storage whenever any external lookup occurred, regardless of why the lookup happened (even incidental lookups for personal/memory queries).

**Additional Causes Found**:
- `orchestrator.js` had a standalone `matchesNewsPattern` condition that triggered external lookup for ANY message containing "today", "news", "yesterday", or geopolitical terms — even "I need emotional support today"
- `truthTypeDetector.js` had no patterns for conversational/personal queries, so "I need support today" was caught by the VOLATILE "today" pattern

## Files Modified

### 1. `api/core/intelligence/hierarchyRouter.js`
**Change**: `getSourceHierarchy()` — changed AMBIGUOUS default from `EXTERNAL_FIRST` to `VAULT_FIRST`

AMBIGUOUS means "unknown intent," not "needs external data." Queries that don't match business policy or objective factual patterns should look in memory/vault first, not trigger external API calls.

### 2. `api/core/intelligence/truthTypeDetector.js`
**Change**: Added `CONVERSATIONAL_PATTERNS` block before the VOLATILE pattern check in `detectByPattern()`

New patterns detect greetings, personal facts ("my dog is Max"), memory commands ("remember that my..."), and emotional support requests ("I'm feeling stressed"). These are classified as `PERMANENT` with `skipExternalLookup: true`, preventing VOLATILE words like "today" from triggering external lookups in personal queries.

### 3. `api/core/orchestrator.js`
**Change**: Removed standalone `matchesNewsPattern` from `shouldLookup` conditions

Previously: `matchesNewsPattern` alone (condition 3) triggered external lookup for any message with "today", "news", "yesterday", or geopolitical terms regardless of truth type. Now external lookup only fires when: (1) VOLATILE, (2) SEMI_STABLE + news patterns, (3) high-stakes domain, (4) router explicitly requires external. Legitimate news queries are still covered since AMBIGUOUS → Stage 2 → SEMI_STABLE, which paired with news patterns triggers condition (2).

### 4. `server.js`
**Change**: Both storage paths (intelligent storage ~line 553 and supersession-aware ~line 619) now check `isPersonalOrMemoryQuery` before skipping storage

The gate now skips storage ONLY when `hasExternal=true AND the message is NOT a personal/memory query`. Personal facts, memory commands, greetings, and self-disclosures are stored even if external lookup fired incidentally (e.g., high-stakes safety verification on an allergy disclosure).

### 5. `tests/tier1/code-guards.test.js`
**Change**: Updated `refLineApprox` in useClaude allowlist from `4151` to `4161`

The 10 comment lines added to orchestrator.js's `shouldLookup` block shifted the catch block `useClaude` reference from line 4151 to 4161. The previous value was within the `< 10` tolerance (|4160-4151|=9), but updating it keeps the allowlist accurate for future changes.
