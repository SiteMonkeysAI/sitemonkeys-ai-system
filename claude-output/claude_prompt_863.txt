## Failure 1: STR1 — Volume Stress (10 Facts, Recall 3)

### What the Test Does

Stores 10 distinct personal facts in a single message:
- Car: Tesla Model 3
- Dog's name
- Favorite color
- Job title
- City
- Hobby
- Sibling count
- College
- Anniversary date
- Morning routine

Then asks for 3 specific facts back (car, dog, color).

### What Happened

Car ✅ recalled correctly. Color ✅ recalled correctly. Dog ❌ not recalled.

### Expected Behavior

All 10 facts should be stored. Any 3 should be retrievable. The system Bible (Innovation #1) specifies 3-6M token capacity. 10 simple facts is approximately 100-200 tokens — well within budget.

### What to Investigate

1. **Was the dog fact extracted?** Check if the GPT-4o-mini extraction step produced a fact line containing the dog's name from the 10-fact input. If not, the extraction prompt may be dropping facts when too many are present in a single message.

2. **Was it stored?** If extracted, did it make it into the database? The aggressive post-processing (`aggressivePostProcessing`) has a fact count cap — if it's set to 3-5 facts, it would drop 5-7 of the 10 input facts.

3. **Was it retrieved?** If stored, did the retrieval query for "dog" find it? The semantic retrieval might not connect "dog" to a fact stored under a different category or with different keywords.

4. **The fact count cap in `aggressivePostProcessing` is the most likely culprit.** If the system limits output to 3-5 facts per message, a 10-fact message will always lose data. For legitimate dense personal fact inputs, the cap needs to be higher or the system needs to detect that all lines are distinct facts (not redundant) and preserve them all.

### Required Fix

When a user provides multiple distinct facts in a single message, ALL of them must be stored. The system should:
- Extract all facts from the message (the extraction prompt should be instructed to preserve all distinct facts, not summarize)
- The post-processing fact count cap should scale with the number of distinct facts detected, not be a fixed number
- At minimum, 10 distinct simple facts in one message should all survive extraction, compression, and storage

## Failure 2: CMP2 — International Name Preservation (Deep Test Specific)

### What the Test Does

Stores names with international characters: Zhang, Björn, José. Then asks for them back.

### What Happened

The test returned the NUA2 response (allergy vs. wife preference) instead of the CMP2 response (international names). The names were not found.

### Important Context

This test PASSES in manual testing. When a real user says "My friend's name is José García and my colleague is Björn Þórsson" and then asks "What are my friends' names?", the system correctly returns both names with accents intact. The compression pipeline fix (PR #843) is working.

The failure is specific to the automated deep test, which creates a fresh test user (`smdeep-XXXXX`) and runs 15 tests in rapid succession.

### What to Investigate

1. **Timing:** Does the extraction/storage complete before the retrieval query fires? The deep test may not wait long enough between storing and querying. If storage is async and the query runs before the INSERT completes, the names won't be found.

2. **Test user context:** The fresh test user has no prior memory. The semantic retrieval might need a minimum amount of context to route queries correctly. With only a few facts stored, the retrieval may not find the right category.

3. **Cross-test contamination:** The test got the NUA2 response (allergy/cats) instead of the CMP2 response. This suggests the retrieval pulled facts from a different test in the same session. The test user isolation may not be working correctly — facts from test NUA2 are being retrieved when CMP2 queries.

4. **Category routing:** The international names might be stored in a different category than where the retrieval query looks. If "My colleagues are Zhang, Björn, and José" routes to `relationships_social` but the query "What are their names?" routes to a different category, the names won't be found.

### Required Fix

The deep test must pass, not just manual testing. The fix needs to ensure:
- Facts stored by the CMP2 test are retrievable by the CMP2 query within the same test user session
- No cross-contamination between test sections (NUA2 facts shouldn't appear in CMP2 responses)
- If timing is the issue, the storage path must confirm completion before the test proceeds
- If category routing is the issue, name-related queries should search across categories, not just one

## Failure 3: EDG3 — Long Input Preservation (Pricing Detail)

### What the Test Does

Sends a long, dense input containing multiple business details about an artisanal product business, including:
- Product descriptions
- Pricing tiers
- Competitive advantages
- Target market
- Distribution strategy

Then asks for specific details back (pricing, competitive advantage).

### What Happened

Competitive advantage ✅ recalled. Pricing ❌ not recalled. The response said "I don't have specific information about the pricing for your artisana..."

### Expected Behavior

The system Bible (Chapter 9.3) states: "Hard questions should yield MORE intelligence, not less." A user sharing their complete business plan is providing critical decision-relevant information. The pricing is arguably the MOST important detail to preserve.

### What to Investigate

1. **Extraction prioritization:** When GPT-4o-mini extracts facts from a long input, does it prioritize correctly? Pricing ($X for Y product) should be high-priority because it contains specific numbers — the extraction prompt should treat numeric facts as higher priority than descriptive text.

2. **Compression truncation:** The `aggressivePostProcessing` word-limit (5 words per line) may be truncating pricing lines like "Artisanal soap bars: $12.99 each, gift sets: $45.99" into "Artisanal soap bars: $12.99" — losing half the pricing data. Lines containing dollar amounts or pricing information should be exempt from word-count truncation, similar to how identifier lines and unicode name lines are already exempt.

3. **Fact count cap (same as STR1):** If the long input generates 15+ fact lines but the cap is 5, most facts are dropped. Pricing might be below the cutoff.

### Required Fix

For dense business inputs:
- Pricing/financial details (lines containing dollar amounts, percentages, or explicit pricing language) must be treated as high-priority facts, exempt from word-count truncation
- The fact count cap must accommodate dense inputs — a user sharing their complete business plan should have all key details preserved, not just the first 3-5
- The extraction prompt should explicitly instruct: "Preserve ALL numeric values, pricing, dates, and deadlines — these are the highest-priority facts"

## The Common Thread

All three failures share a root cause: **the system drops legitimate facts when inputs are dense.** The recent storage pipeline hardening (zombie cleanup, interrogative filtering, aggressive compression) correctly prevents garbage from entering the database. But the aggressiveness settings are also preventing legitimate dense inputs from being fully preserved.

The balance point: keep all the pollution filters (they're working) but increase the preservation capacity for legitimate fact-dense messages. The system should be able to distinguish between:
- A question asking for recall (skip storage) ← working correctly
- System metadata (block storage) ← working correctly  
- A single message containing 10 personal facts (store ALL of them) ← currently dropping some
- A long business plan with pricing details (store pricing as priority) ← currently dropping pricing
