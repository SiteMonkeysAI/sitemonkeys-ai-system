## Root Cause

Weight-based commodity queries like "what is 50 lbs of gold worth" were returning generic non-answers because they were being misclassified as `PERMANENT` truth type instead of `VOLATILE`, causing the external lookup to be skipped entirely.

**Execution path break point:**
- `detectByPattern()` in `truthTypeDetector.js` matched the query against PERMANENT pattern `/\b(what is|define|...)\b/i` (matched "what is")
- The VOLATILE patterns only had `/\bhow much (is|does|are) .* (cost|worth)\b/i` which requires "how much" prefix — not "what is" prefix
- Since `truthTypeResult.type = PERMANENT`, the orchestrator's `shouldLookup = false` and `lookup()` was never called

The `metals.live` source, weight conversion math, and `hasValueIntent` guard were all correct. The failure was 100% in truth type classification — the query never reached the commodity lookup.

## Files Modified

### 1. `api/core/intelligence/truthTypeDetector.js`
Added a new VOLATILE pattern to cover "what is/are/what's X worth" queries:
```javascript
/\bwhat (is|are|'?s) .{0,60} worth\b/i,
```
This catches "what is 50 lbs of gold worth", "what's 2 kg of silver worth", and similar price-intent queries that use "what is" phrasing instead of "how much is". VOLATILE wins over the existing PERMANENT match due to conflict resolution logic.

### 2. `api/core/intelligence/externalLookupEngine.js`
Added "worth" to `FRESHNESS_MARKERS` price-intent pattern:
```javascript
/\b(price|stock|rate|value|cost|worth)\b/i,  // was: price|stock|rate|value|cost
```
This ensures `isLookupRequired()` also triggers on "worth" as a secondary defense, even if truth type classification ever misses it.
